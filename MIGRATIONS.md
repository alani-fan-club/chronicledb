# Migrations

ChronicleDB does not use a formal migration framework. The plugin is redeployed in-place against existing live databases, so every schema change must be forward-compatible with older deployments. The canonical pattern is idempotent `CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, and `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` statements in `server-plugin/schema.sql`, executed on plugin boot.

## Pattern

When evolving a table:

1. Add an `ALTER TABLE <name> ADD COLUMN IF NOT EXISTS <col> <type> [DEFAULT ...];` line to `schema.sql`, located immediately after the table's original `CREATE TABLE` block so the column appears on both fresh installs (via the CREATE) and existing installs (via the ALTER).
2. Keep the column nullable, or supply a `DEFAULT`, so the `ALTER` succeeds against rows already in the table.
3. If a backfill is required (e.g. recomputing embeddings), add a dedicated script under `server-plugin/` and document how to run it alongside the ALTER.
4. Never drop a column in place without first confirming no in-flight deployment still writes it. Prefer adding a replacement column and flipping readers, then removing the old column in a follow-up.

New tables follow the same rule: use `CREATE TABLE IF NOT EXISTS`. New indexes use `CREATE INDEX IF NOT EXISTS`. On plugin boot, `schema.sql` is executed top-to-bottom and any statement that is already applied is a no-op.

- `items.chat_id` â€” `ALTER TABLE items ADD COLUMN IF NOT EXISTS chat_id TEXT`. Scopes `OWNS` / `LOCATED_AT` edges to a single chat. Content-addressed id now includes `chat_id` so the same-named item in two chats becomes two rows. Legacy NULL-chat rows are dropped from chat-scoped queries; re-ingest to populate.
- `knows.chat_id` â€” `ALTER TABLE knows ADD COLUMN IF NOT EXISTS chat_id TEXT`. Scopes `KNOWS` edges so a character can know a fact in chat A without leaking it into chat B. - `present_at.chat_id` â€” `ALTER TABLE present_at ADD COLUMN IF NOT EXISTS chat_id TEXT`. Scopes character presence to a chat so extraction in chat A doesn't flip `is_current=FALSE` on rows from chat B.
- `traits.normalized_content` â€” generated `STORED` column that lowercases `content` and strips non-alphanumeric. Backed by a UNIQUE index on `(character_id, category, normalized_content)` so `upsertTrait` dedupes case/punctuation variants ("Awed" / "awed" / "Awe-struck" / "Awestruck") via `ON CONFLICT DO NOTHING`. One-time cleanup deletes pre-existing duplicates keeping the oldest row per normalized group.
- `traits.stemmed_content` â€” generated `STORED` column from `strip(to_tsvector('english', content))::text`. Runs the Postgres English stemmer so `charmed` / `charming` / `charms` all collapse to `'charm'`, `observant` / `observing` to `'observ'`, etc. `upsertTrait`'s fuzzy pre-check compares against this to catch morphological variants the exact-normalized index misses. Non-unique index (not UNIQUE) so legitimate multi-word traits with overlapping stems can coexist. One-time cleanup kept the longest content per `(character, category, stemmed_content)` group.
- `traits.embedding` â€” `ALTER TABLE traits ADD COLUMN IF NOT EXISTS embedding vector(768)`. Per-trait contextual embedding populated lazily by the extractor (Path 1 contextual embeddings). Nullable: older rows stay NULL until re-ingested. Feeds the per-character rollup in `characters.summary_embedding` â€” `recomputeCharacterSummary` skips rows where `embedding IS NULL`.
- `characters.summary_embedding` â€” `ALTER TABLE characters ADD COLUMN IF NOT EXISTS summary_embedding vector(768)`. Per-character mean-pool rollup of all `personality`-category trait embeddings, computed via pgvector's native `AVG(vector)`. Recomputed by `db.recomputeCharacterSummary` on trait insert/delete; backfilled across all characters via `POST /api/plugins/chronicle-db/recompute-character-summaries`. - `traits.evidence_sentence` â€” `ALTER TABLE traits ADD COLUMN IF NOT EXISTS evidence_sentence TEXT`. One-sentence verbatim quote from the source batch that justifies the trait; becomes the tail half of the contextual embedding text (`"${character} is ${content}: ${evidence_sentence}"`) per Anthropic's contextual retrieval pattern. Nullable â€” legacy rows embed as `"${character} is ${content}"` alone.
- `traits.canonical_id` â€” `ALTER TABLE traits ADD COLUMN IF NOT EXISTS canonical_id TEXT REFERENCES traits(id) ON DELETE SET NULL`. Self-referential pointer to the canonical trait row when this row is a merged variant; `NULL` means this row IS the canonical. On canonical deletion the reference nulls out rather than cascading so the raw-extraction provenance survives.
- `traits.aliases` â€” `ALTER TABLE traits ADD COLUMN IF NOT EXISTS aliases TEXT[] DEFAULT '{}'`. - `traits.merged_count` â€” `ALTER TABLE traits ADD COLUMN IF NOT EXISTS merged_count INT DEFAULT 1`. Number of extraction candidates (including the original) that have been folded into this canonical row. Cheap popularity signal for ranking.
- `idx_traits_canonical` â€” `CREATE INDEX IF NOT EXISTS idx_traits_canonical ON traits (character_id, canonical_id)`. Speeds up the "get all merged variants for a canonical id" lookup and the user-visible `WHERE canonical_id IS NULL` filter per character.
